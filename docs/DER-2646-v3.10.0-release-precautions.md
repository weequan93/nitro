## Critical WASM Precaution

The preserved DER machines from `dev-DER-2646` are still downloaded as:

```text
machine.wavm.br
```

The v3.10 validator default expects:

```text
machine.v2.wavm.br
```

Before production release, confirm the validator image can load every required machine root. Do not activate ArbOS 60 until this is proven.

Acceptable resolutions:

- Provide DER-compatible `machine.v2.wavm.br` artifacts for old DER roots.
- Keep old DER validation served by a legacy validator that supports `machine.wavm.br`.
- Confirm the split validator setup loads both old and new roots without missing-machine errors.

## Pending v2 Validation Before ArbOS 60

Pending validation is a local validator/staker check. It is not a contract transaction and it does not activate ArbOS 60.

Use it to prove the new v3.10 `machine.v2.wavm.br` root can replay the current live ArbOS 32 chain before the rollup officially switches to that root.

Example configuration shape:

```text
--node.block-validator.enable=true
--node.block-validator.current-module-root=current
--node.block-validator.pending-upgrade-module-root=$NEW_WASM_MODULE_ROOT
```

Meaning:

```text
current-module-root=current
  Validate with the current on-chain L1 rollup wasmModuleRoot.

pending-upgrade-module-root=$NEW_WASM_MODULE_ROOT
  Also validate the same live blocks with the future v2 machine root.
```

This does not:

- Change the L1 rollup `wasmModuleRoot`.
- Make new assertions officially use the v2 root.
- Activate ArbOS 60.
- Prove ArbOS 60 behavior after activation.

This does prove:

- The new v2 machine can load.
- The new v2 validator can execute current ArbOS 32 blocks.
- The new v2 validation run reaches the same expected final global state as the current chain execution.

The success condition is:

```text
old/current root validates block N successfully
new v2 pending root validates block N successfully
both reach the expected final GlobalState for block N
```

Treat pending validation as required before calling `setWasmModuleRoot($NEW_WASM_MODULE_ROOT)`.

After the v2 root is official, continue monitoring validators while ArbOS is still 32. Only schedule ArbOS 60 after the official v2 root has validated cleanly.

## Smart Contract Actions

There are two separate controls:

1. L1 rollup WASM root
2. L2 ArbOS activation schedule

The L1 rollup WASM root controls assertion/proving compatibility.

Read current root:

```bash
cast call \
  --rpc-url $L1_RPC \
  $ROLLUP_ADDRESS \
  "wasmModuleRoot()(bytes32)"
```

Update root, only if the new release requires a new proving machine:

```bash
cast send \
  --rpc-url $L1_RPC \
  --private-key $ROLLUP_OWNER_PRIVATE_KEY \
  $ROLLUP_ADDRESS \
  "setWasmModuleRoot(bytes32)" \
  $NEW_WASM_MODULE_ROOT
```

Confirm the root after update:

```bash
cast call \
  --rpc-url $L1_RPC \
  $ROLLUP_ADDRESS \
  "wasmModuleRoot()(bytes32)"
```

Do not update the L1 WASM root to a value not included in the deployed validator images.

## Activation Control

ArbOS activation is controlled by the L2 `ArbOwner` precompile:

```text
ArbOwner: 0x0000000000000000000000000000000000000070
```

Schedule ArbOS 60:

```bash
cast send \
  --rpc-url $L2_RPC \
  --private-key $CHAIN_OWNER_PRIVATE_KEY \
  0x0000000000000000000000000000000000000070 \
  "scheduleArbOSUpgrade(uint64,uint64)" \
  60 \
  $ACTIVATION_TIMESTAMP
```

Activation happens at the first L2 block where:

```text
block.timestamp >= ACTIVATION_TIMESTAMP
```

Do not use timestamp `0` in production unless immediate next-block activation is intended.

Recommended activation window:

```text
ACTIVATION_TIMESTAMP = current time + 2 to 6 hours
```

Before activation, the schedule can be changed by calling:

```text
scheduleArbOSUpgrade(60, NEW_TIMESTAMP)
```

After activation, ArbOS cannot be downgraded by schedule.

## Rollout Order

1. Prepare and tag immutable release artifacts.
2. Confirm the working tree and submodules are clean or intentionally documented.
3. Confirm current L1 `wasmModuleRoot`.
4. Confirm no unexpected L2 ArbOS upgrade is already scheduled.
5. Upgrade passive full/RPC nodes first.
6. Upgrade validators/stakers with the validator image and both legacy/new root paths.
7. Confirm validators can load the current on-chain WASM root and the new v2 root.
8. Configure pending validation with `pending-upgrade-module-root=$NEW_WASM_MODULE_ROOT`.
9. Confirm pending v2 validation passes against live ArbOS 32 blocks.
10. Update L1 `wasmModuleRoot` if required by the release.
11. Confirm new assertions validate correctly with the official v2 root while ArbOS is still 32.
12. Upgrade batch poster and sequencer.
13. Schedule ArbOS 60 activation with a future timestamp.
14. Monitor until activation block.
15. Confirm ArbOS version after activation.
16. Run DER feature checks after activation.
17. Keep old images and machines available until the challenge/dispute window is safely passed.

## Activation Monitoring

Before activation:

- Full nodes follow the same head.
- Sequencer produces blocks normally.
- Batch poster posts batches normally.
- Validator logs show no missing WASM root errors.
- If the L1 root will be changed, pending v2 validation has passed on live ArbOS 32 blocks.
- If the L1 root has already been changed, new assertions validate correctly with the official v2 root.
- `getScheduledUpgrade()` returns the expected version and timestamp.

During activation:

- Watch the first L2 block with `block.timestamp >= ACTIVATION_TIMESTAMP`.
- Confirm the sequencer keeps producing blocks.
- Confirm full/RPC nodes do not diverge.
- Confirm validators/stakers do not fail machine loading.

After activation:

```bash
cast call \
  --rpc-url $L2_RPC \
  0x0000000000000000000000000000000000000064 \
  "arbOSVersion()(uint256)"
```

Expected for internal ArbOS 60:

```text
115
```

## Things Not To Change During This Release

Do not change these unless there is a separate tested migration plan:

- DER storage slots
- DER precompile addresses
- DER precompile ABI surface
- production `chainId`
- rollup contract address
- bridge address
- inbox address
- sequencer inbox address
- genesis block number
- chain owner
- serialized chain config
- `MaxUncompressedBatchSize`

Avoid calling `setChainConfig` during this release unless the serialized chain config has been tested end to end. In v3.10, `MaxUncompressedBatchSize` is part of config compatibility checks, so accidental changes can cause node incompatibility.

## Rollback Notes

Before ArbOS 60 activation:

- Node image rollback is possible if the chain has not passed the scheduled activation timestamp.
- The scheduled timestamp can be moved later by the chain owner.

After ArbOS 60 activation:

- Do not assume protocol rollback is possible.
- Keep old and new validator machines available.
- Keep the previous image tags available for non-protocol operational rollback only.

## Final Go / No-Go

Go only if all are true:

- Release branch and submodules are clean or intentionally documented.
- Current and next WASM roots are present and loadable by validators.
- Pending v2 validation has passed before changing the L1 `wasmModuleRoot`.
- The official v2 root validates cleanly while ArbOS is still 32 before scheduling ArbOS 60.
- L1 `wasmModuleRoot` is correct.
- No unexpected ArbOS upgrade is scheduled.
- All node roles are upgraded before activation.
- Chain owner key is available for scheduling or delaying activation.
- Rollup owner key is available if `setWasmModuleRoot` is required.
- Existing DER state reads correctly before activation.
- DER checks pass after activation on staging or DB copy.
